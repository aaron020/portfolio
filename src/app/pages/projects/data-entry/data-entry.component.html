Project Overview

Basically, a local business was in desperate need of a new data entry system—and when I say desperate, I mean their current system was running on a Windows XP computer. A computer they were afraid to turn off in case it didn’t turn back on. It was just an old database with a basic, almost Excel-looking UI.


One of their major concerns was that they had never been able to back up any of their data. So that was the first thing I tackled—it took a bit of digging, but I eventually found a way to convert the database into a CSV file (thankfully). Once I had the data in CSV format, I worked on migrating it into a more modern, reliable database — one that could be easily backed up and restored. So, I set up a MySQL database.


Now came the issue of making a UI to front this database. I spoke with the owner of the business—he had a lot of ideas for what he wanted the UI to do, new features that should be added, and what needed to be kept from the old UI. Tbh, I knew it would be easy enough to impress when their previous UI was basically an Excel sheet.

How it works

The frontend is just HTML/ CSS with bootstrap. I used python flask for the backend, then a MySQL database. They did not want it online so the application and database run locally (they backup the database reguarly)

Features added and why


Backup & Restore

So, I've mentioned this a few times now, but one of their major issues was not being able to back up the database. So I wrote a Python script that uses the mysqldump.exe program to back up the database to a specified file location. I also stored backup information in the database, allowing them to see when their last backup was, what the file was called, and where the backup was stored.

And obviously there wouldnt be much point being able to backup, if you couldnt also restore from backup... So I added that also


The actual data entry / document generation

As this is a data entry system, the main feature here is obviously being able to actually enter data. But if you look down at the bottom of the screenshot, users can also generate several different business documents based on the data they've already entered, such as invoices, work orders, delivery dockets, etc.

Heres what a generated invoice looks like (all data added automatically, calculation done automatically)

Another annoying task they faced was collecting all invoices for each customer dealt with during the month and printing them individually at month-end. The owner mentioned this process would take an employee for half a day, sometimes even a full day. To address this, I created an automated solution, the data was all there in the database - it just meant timestamping, pulling data for the correct customer and generating the invoices (which I had already done). Now, they simply need to enter the date range and customer number to generate a list of invoices, hit print, done.


Customer search

During data entry, they struggled to locate customer numbers for specific customers. To solve this problem, I developed a search feature that allowed them to find customers by entering a full name or just a few initial letters. With the search results, they could simply click an "add" button to include the customer in their current data entry session, or make edits to the customer information if needed.

What I learnt / Conclusion

I learnt a lot working on this project. It was interesting to work directly with the customer that would be using said project — converting their requirements into actual working features was difficult at times (as their requirements would change occasionally), but it was also very rewarding to see something I had built being used in the real world to actually help and save people a lot of time.
